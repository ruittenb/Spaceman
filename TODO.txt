
================================================================================
DECORATIONS: allow classes of icons to have different styling
================================================================================

Status:   PLANNING (2026-03-26)
Effort:   Medium — touches IconCreator, PreferencesView, new model type,
          settings migration
Risk:     Low — rendering-only change, no persistence or observer impact.
          Settings migration needed for existing users.

--- What it does ---

Each icon currently gets the same shape (rounded rect with cornerRadius 3).
The only visual distinction is active-vs-inactive (filled/dimmed or outlined).
This feature lets the user pick a decoration (shape + fill) independently for
four classes of icons:

  - active space
  - inactive space
  - fullscreen active space
  - fullscreen inactive space

Example: active = filled pill, inactive = rounded outline, fullscreen = square.

--- The two axes ---

Shape (mutually exclusive per icon class):
  - borderless  — no box at all, just the text (cf. bare-names branch)
  - square      — sharp corners, cornerRadius = 0
  - rounded     — small rounding, cornerRadius = Constants.boxCornerRadius (3.0)
  - pill        — fully rounded, cornerRadius = boxRect.height / 2
                   (cf. round-icons branch)

Fill (mutually exclusive per icon class):
  - filled      — solid background; text knocked out (template) or contrasting
  - outline     — border stroke only; text drawn inside

"Borderless" ignores the fill axis — there's no shape to fill or outline.

Fullscreen spaces currently force cornerRadius = 0 regardless of decoration.
With this feature, the user explicitly picks the fullscreen decoration, so
the fullscreen override in IconCreator would read from the new setting
rather than hardcoding square.

--- Prior art in the codebase ---

bare-names branch:
  - Added DisplayStyle.bareNumbers / .bareNames → text without a box
  - isBareStyle computed property on DisplayStyle
  - When bare: gapWidth = 0, fontSize += 2
  - This is equivalent to "borderless" decoration. With decorations, the
    bare display styles would no longer be needed — DisplayStyle goes back
    to controlling text content only, and borderless is a decoration choice.

round-icons branch:
  - Changed cornerRadius from Constants.boxCornerRadius to boxRect.height / 2
  - This is equivalent to the "pill" decoration.

Both branches prove the rendering changes are small and isolated.

--- What replaces what ---

InactiveStyle (dimmed / bordered) gets absorbed:
  - "Dimmed" = active and inactive both use the same shape, but inactive
    is at reduced opacity. This is now: same decoration for both, plus
    the existing Constants.inactiveAlpha.
  - "Bordered" = active is filled, inactive is outline. This is now:
    active decoration = filled, inactive decoration = outline.
  InactiveStyle.swift can be removed once decorations are in place.

DisplayStyle.numbers ("Bare Numbers") gets absorbed:
  - "Bare Numbers" = numbers text + borderless decoration.
  - With decorations, DisplayStyle simply controls what text is shown
    (none / numbers / names / numbers+names). The shape is a separate
    concern. The numbersAndRects case also simplifies to just "numbers"
    since the "rects" part is implied by the decoration being non-borderless.

Summary of DisplayStyle simplification:
  Current:                     After decorations:
  .rects          (no text)    .rects     (no text)
  .numbers        (bare nums)  .numbers   (was numbersAndRects)
  .numbersAndRects (nums+box)  .names
  .names                       .numbersAndNames
  .numbersAndNames

--- Settings model ---

New enum Decoration, stored as an Int for AppStorage:

  enum Decoration: Int, CaseIterable {
      case borderless = 0
      case squareOutline = 1
      case squareFilled = 2
      case roundedOutline = 3
      case roundedFilled = 4
      case pillOutline = 5
      case pillFilled = 6
  }

Collapsing shape + fill into a single enum keeps the number of AppStorage
keys manageable (4 keys instead of 8) and makes the preferences UI simpler
(one picker per icon class instead of two).

AppStorage keys:
  "decorationActive"            default: .roundedFilled
  "decorationInactive"          default: .roundedOutline
  "decorationFullscreenActive"  default: .squareFilled
  "decorationFullscreenInactive" default: .squareOutline

These defaults match today's behavior (rounded filled/outline for normal
spaces, square for fullscreen) so existing users see no change.

--- Preferences UI ---

Replace the current "Inactive style" segmented control with a decoration
section. Two layout options:

  Option A — Four pickers (explicit, verbose):

    Active space         [Rounded, filled  ▼]
    Inactive space       [Rounded, outline ▼]
    Fullscreen active    [Square, filled   ▼]
    Fullscreen inactive  [Square, outline  ▼]

  Option B — Two pickers + a fullscreen toggle (compact):

    Active space         [Rounded, filled  ▼]
    Inactive space       [Rounded, outline ▼]
    ☐ Same for fullscreen (but square)

    When checked: fullscreen copies the fill choice but forces square.
    When unchecked: shows two additional pickers for fullscreen.

  Recommendation: start with Option A. Four pickers is straightforward,
  and the UI can always be collapsed later if it feels cluttered.

  The picker labels:

    Borderless         (= just text, no box)
    Square, outline
    Square, filled
    Rounded, outline
    Rounded, filled
    Pill, outline
    Pill, filled

--- Rendering changes (IconCreator.swift) ---

createSpaceIcon() currently branches on isBareNumbers (→ text only) vs
everything else (→ box rendering). The refactored flow:

  1. Determine icon class: active/inactive × normal/fullscreen
  2. Look up Decoration for that class from AppStorage
  3. If borderless:
       - Draw text only (existing bare-numbers path)
       - Apply bare-style adjustments: gapWidth = 0, fontSize += 2
  4. If filled:
       - Compute cornerRadius from shape (0 / boxCornerRadius / height÷2)
       - Draw filled box + text (existing active/dimmed path)
  5. If outline:
       - Compute cornerRadius from shape
       - Draw stroked outline + text (existing bordered path)

createMenuItemIcon() is unaffected — it always draws filled boxes for
the right-click menu.

The fullscreen cornerRadius override
(line 161: `let cornerRadius = space.isFullScreen ? 0.0 : ...`)
is replaced by reading the fullscreen decoration's shape.

--- Borderless adjustments ---

From the bare-names branch: when borderless, gapWidth and displayGapWidth
should be 0 (icons touch), and fontSize should be slightly larger (+2).

But with per-class decorations, some icons in the row might be borderless
and others not. Two approaches:

  a) Only apply the gap/font adjustment when ALL icons use borderless.
     Simple, but mixed decorations look odd with normal gaps around
     bare text.

  b) Per-icon gap adjustment: each icon gets gapWidth = 0 on its side
     if it is borderless. Font size bumps per-icon based on its
     decoration. This is the right approach because the gap/font
     is about the individual icon, not the whole row.

     Implementation: pass the decoration into createSpaceIcon() and
     let it adjust its own metrics. The merging functions
     (mergeIcons / mergeIconsTwoRows) already handle variable-width
     icons, so different per-icon widths are fine.

  Recommendation: (b). It requires the gap to be determined per-icon
  rather than once in getIcon(), but this is a small change.

--- Settings migration ---

Existing users have:
  - "displayStyle" with value 1 (bare numbers) or 2 (numbers+rects)
  - "inactiveStyle" with value 0 (bordered) or 1 (dimmed)

Migration (one-time, on launch):
  1. If "displayStyle" == 1 (bare numbers):
       Set decorationActive = .borderless
       Set decorationInactive = .borderless
       Set new displayStyle = .numbers (was numbersAndRects)
  2. If "inactiveStyle" == 0 (bordered):
       Set decorationInactive = .roundedOutline (default, but explicit)
     If "inactiveStyle" == 1 (dimmed):
       Set decorationInactive = .roundedFilled
       (Dimmed = same shape as active, just lower opacity. The opacity
       is already applied by Constants.inactiveAlpha when active and
       inactive both use a filled decoration.)
  3. Remove the "inactiveStyle" key.
  4. Guard with a "decorationsMigrated" bool key.

--- Dimmed rendering ---

Today, "dimmed" means: same filled box as active, but at
Constants.inactiveAlpha (0.4). With decorations, the dimmed behavior is
implicit: if active = roundedFilled and inactive = roundedFilled, the
inactive icon uses the same rendering path but applies inactiveAlpha.

The alpha logic (isActive ? 1.0 : Constants.inactiveAlpha) already exists
in createSpaceIcon(). It just needs to check if the decoration is any
filled variant, rather than checking inactiveStyle == .dimmed.

--- Future-proofing ---

The Decoration enum is easily extensible. If more shapes are wanted later
(e.g., diamond, hexagon), add a case and a cornerRadius/path computation.
The per-class AppStorage pattern scales to additional icon classes if
needed (e.g., "current display active" vs "other display active").

================================================================================
OPTION-HOVER: show extra info on the menu bar icon
================================================================================

Status:   PARKED (2026-03-25)
Effort:   Low — isolated change, entirely within StatusBar.swift
Risk:     None — additive, no architectural impact, easy to remove

Why parked: The itch that prompted this isn't strong right now. The mechanism
is simple enough to add whenever it's needed, and the "what to show" decision
doesn't need to be made upfront.

--- What it does ---

When the user hovers over the menu bar icon while holding Option, show
additional detail about the spaces.

--- What to show (candidates) ---

  a) Space numbers and/or names — for users who use compact/number styles and
     occasionally want to see which space is which without opening the menu.
     Easiest.

  b) Space screenshots — tiny thumbnails of each space's content. Much cooler
     but much harder: needs CGWindowListCreateImage, Screen Recording
     permission, and image rendering. Different league of effort.

  c) Other ideas considered: focused app per space.

--- How to present it (candidates) ---

  a) Tooltip — simplest. Just set statusBarItem.button.toolTip to a string.
     Plain text only, no formatting control. Only works for (a) above.

  b) NSPopover — speech balloon like the help popovers in Preferences (which
     use SwiftUI .popover). The menu bar icon is AppKit, so this would use
     NSPopover with an NSHostingView to embed SwiftUI content. More effort
     but can show rich/formatted content, and the pattern already exists in
     the codebase (PreferencesView.swift:262, :421). Works for all options.

--- Hover detection (common to all approaches) ---

All in StatusBar.swift (~15-20 lines):

  1. NSTrackingArea on statusBarItem.button for mouseEntered/mouseExited
  2. On mouseEntered: check NSEvent.modifierFlags.contains(.option)
  3. If Option held → show the detail (tooltip or popover)
  4. On mouseExited → hide it

  let area = NSTrackingArea(
      rect: button.bounds,
      options: [.mouseEnteredAndExited, .activeAlways, .inVisibleRect],
      owner: self
  )
  button.addTrackingArea(area)

  Notes:
  - .activeAlways works even when the app isn't frontmost (menu bar item)
  - .inVisibleRect auto-updates the rect if the button resizes
  - mouseEntered/mouseExited go to the owner (StatusBar inherits from NSObject)

--- Future-proofing ---

No concerns. The tracking area + event monitor pattern is standard AppKit.
If more modifier-based behaviors are added later (Cmd-hover, etc.), the
flag-checking can be refactored trivially at that point.
