
================================================================================
DECORATIONS: allow classes of icons to have different styling
================================================================================

Status:   PLANNING (2026-03-27)
Effort:   Medium — touches IconCreator, PreferencesView, new model type,
          settings migration
Risk:     Low — rendering-only change, no persistence or observer impact.
          Settings migration needed for existing users.

--- What it does ---

Each icon currently gets the same shape (rounded rect with cornerRadius 3).
The only visual distinction is active-vs-inactive (filled/dimmed or outlined).
This feature lets the user pick a decoration (shape + fill) independently for
active and inactive spaces:

  - active space decoration
  - inactive space decoration

Fullscreen spaces automatically use a rectangular shape — unless the user's
regular decoration is already rectangular, in which case fullscreen uses pill
(so there's always a visual distinction).

Example: active = pill filled, inactive = rounded border.

--- The decoration options ---

A single axis combining shape and fill:

  bare text            — no box at all, just the text
  rectangular, border  — sharp corners, outline only
  rectangular, filled  — sharp corners, solid fill
  rounded, border      — cornerRadius 3px, outline only
  rounded, filled      — cornerRadius 3px, solid fill
  pill, border         — cornerRadius = height/2, outline only
  pill, filled         — cornerRadius = height/2, solid fill

"Bare text" is a special case that has no shape and therefore no fill.
The shape/fill distinction is only meaningful for the other six options.

--- Dimming rule ---

Inactive icons are rendered at full opacity, UNLESS they have the same
decoration as active icons — in which case Constants.inactiveAlpha (0.4)
kicks in as a fallback so they remain visually distinguishable.

This means:
  - Different decorations (the common case): the shape/fill difference
    IS the distinction. No dimming. Everything renders crisp.
  - Same decoration (e.g., both rounded filled): dimming kicks in
    automatically, giving the same behavior as today's "dimmed" style.
  - Both bare text: inactive dims to 40%.

The rule is a simple equality check: activeDecoration == inactiveDecoration.

--- Fullscreen shape rule ---

No separate fullscreen decoration pickers. Instead, fullscreen spaces
use an automatic shape:

  - If the space's decoration shape is NOT rectangular → fullscreen
    uses rectangular (current behavior, just derived automatically).
  - If the space's decoration shape IS rectangular → fullscreen uses
    pill (so fullscreen always looks different from regular spaces).

The fill (border vs filled) is preserved from the space's decoration.
Only the shape changes for fullscreen.

This keeps the UI to two pickers (active + inactive) while still
visually distinguishing fullscreen spaces.

--- Prior art in the codebase ---

bare-names branch:
  - Added DisplayStyle.bareNumbers / .bareNames → text without a box
  - isBareStyle computed property on DisplayStyle
  - When bare: gapWidth = 0, fontSize += 2
  - This is equivalent to "bare text" decoration. With decorations,
    the bare display styles are no longer needed — DisplayStyle goes
    back to controlling text content only.

round-icons branch:
  - Changed cornerRadius from Constants.boxCornerRadius to
    boxRect.height / 2
  - This is equivalent to the "pill" decoration.

Both branches prove the rendering changes are small and isolated.

--- What replaces what ---

InactiveStyle (dimmed / bordered) gets absorbed:
  - "Bordered" = active is filled, inactive is border. Now expressed
    directly by picking filled for active and border for inactive.
  - "Dimmed" = active and inactive use the same decoration; the dimming
    rule (see above) automatically applies Constants.inactiveAlpha.
  InactiveStyle.swift can be removed.

DisplayStyle.numbers ("Bare Numbers") gets absorbed:
  - "Bare Numbers" = numbers text + bare text decoration.
  - With decorations, DisplayStyle simply controls what text is shown.
    The numbersAndRects case simplifies to just "numbers" since the
    box is controlled by the decoration.

Summary of DisplayStyle simplification:
  Current:                     After decorations:
  .rects          (no text)    .rects     (no text)
  .numbers        (bare nums)  .numbers   (was numbersAndRects)
  .numbersAndRects (nums+box)  .names
  .names                       .numbersAndNames
  .numbersAndNames

--- Settings model ---

New enum Decoration, stored as an Int for AppStorage:

  enum Decoration: Int, CaseIterable {
      case bareText = 0
      case rectangularBorder = 1
      case rectangularFilled = 2
      case roundedBorder = 3
      case roundedFilled = 4
      case pillBorder = 5
      case pillFilled = 6
  }

Computed properties on Decoration:

  var isBareText: Bool
  var isFilled: Bool
  var isBorder: Bool
  var cornerRadius: (NSRect) -> CGFloat
      // returns 0 / boxCornerRadius / rect.height÷2 based on shape

  var fullscreenVariant: Decoration
      // swaps rectangular ↔ pill (keeping border/filled),
      // returns self for bareText

AppStorage keys:
  "decorationActive"    default: .roundedFilled
  "decorationInactive"  default: .roundedBorder

These defaults match today's behavior (filled active, bordered inactive,
both with rounded corners) so existing users see no change.

--- Preferences UI ---

Replace the current "Inactive style" segmented control with two
decoration pickers:

  Active style       [Rounded, filled ▼]
  Inactive style     [Rounded, border ▼]

The picker labels:

  Bare text
  Rectangular, border
  Rectangular, filled
  Rounded, border
  Rounded, filled
  Pill, border
  Pill, filled

Seven entries per dropdown — manageable, and the shape names cluster
them into scannable groups.

--- Rendering changes (IconCreator.swift) ---

createSpaceIcon() currently branches on isBareNumbers (→ text only) vs
everything else (→ box rendering). The refactored flow:

  1. Determine decoration for this space:
       - Is it active or inactive? → pick decorationActive or
         decorationInactive from AppStorage
       - Is it fullscreen? → apply decoration.fullscreenVariant
  2. Determine dimming:
       - If inactive AND decorationActive == decorationInactive → dim
       - Otherwise → full opacity
  3. If bareText:
       - Draw text only (existing bare-numbers path)
       - Apply bare-style adjustments: fontSize += 2
  4. If filled:
       - Compute cornerRadius via decoration.cornerRadius(boxRect)
       - Draw filled box + text (existing active/dimmed path)
       - Apply dimming alpha if needed (step 2)
  5. If border:
       - Compute cornerRadius via decoration.cornerRadius(boxRect)
       - Draw stroked outline + text (existing bordered path)
       - Apply dimming alpha if needed (step 2)

createMenuItemIcon() is unaffected — it always draws filled boxes for
the right-click menu.

The fullscreen cornerRadius override
(line 161: `let cornerRadius = space.isFullScreen ? 0.0 : ...`)
is replaced by the fullscreenVariant logic on the Decoration enum.

--- Borderless adjustments ---

From the bare-names branch: when bare text, gapWidth and displayGapWidth
should be 0 (icons touch), and fontSize should be slightly larger (+2).

With per-class decorations, active and inactive icons in the same row
could have different decorations (e.g., active = pill filled, inactive =
bare text). The gap/font adjustments must be per-icon:

  - Pass the decoration into createSpaceIcon()
  - Each icon adjusts its own font size based on its decoration
  - The merging functions (mergeIcons / mergeIconsTwoRows) already
    handle variable-width icons, so different per-icon widths are fine
  - Gap between icons: use 0 when BOTH adjacent icons are bare text,
    normal gap otherwise. (This avoids bare text crowding boxed icons.)

--- Settings migration ---

Existing users have:
  - "displayStyle" with value 1 (bare numbers) or 2 (numbers+rects)
  - "inactiveStyle" with value 0 (bordered) or 1 (dimmed)

Migration (one-time, on launch):
  1. If "displayStyle" == 1 (bare numbers):
       Set decorationActive = .bareText
       Set decorationInactive = .bareText
       Set new displayStyle = .numbers (was numbersAndRects)
  2. If "inactiveStyle" == 0 (bordered):
       Set decorationInactive = .roundedBorder (default, but explicit)
     If "inactiveStyle" == 1 (dimmed):
       Set decorationInactive = .roundedFilled
       (Same as active → dimming rule kicks in automatically.)
  3. Remove the "inactiveStyle" key.
  4. Guard with a "decorationsMigrated" bool key.

================================================================================
OPTION-HOVER: show extra info on the menu bar icon
================================================================================

Status:   PARKED (2026-03-25)
Effort:   Low — isolated change, entirely within StatusBar.swift
Risk:     None — additive, no architectural impact, easy to remove

Why parked: The itch that prompted this isn't strong right now. The mechanism
is simple enough to add whenever it's needed, and the "what to show" decision
doesn't need to be made upfront.

--- What it does ---

When the user hovers over the menu bar icon while holding Option, show
additional detail about the spaces.

--- What to show (candidates) ---

  a) Space numbers and/or names — for users who use compact/number styles and
     occasionally want to see which space is which without opening the menu.
     Easiest.

  b) Space screenshots — tiny thumbnails of each space's content. Much cooler
     but much harder: needs CGWindowListCreateImage, Screen Recording
     permission, and image rendering. Different league of effort.

  c) Other ideas considered: focused app per space.

--- How to present it (candidates) ---

  a) Tooltip — simplest. Just set statusBarItem.button.toolTip to a string.
     Plain text only, no formatting control. Only works for (a) above.

  b) NSPopover — speech balloon like the help popovers in Preferences (which
     use SwiftUI .popover). The menu bar icon is AppKit, so this would use
     NSPopover with an NSHostingView to embed SwiftUI content. More effort
     but can show rich/formatted content, and the pattern already exists in
     the codebase (PreferencesView.swift:262, :421). Works for all options.

--- Hover detection (common to all approaches) ---

All in StatusBar.swift (~15-20 lines):

  1. NSTrackingArea on statusBarItem.button for mouseEntered/mouseExited
  2. On mouseEntered: check NSEvent.modifierFlags.contains(.option)
  3. If Option held → show the detail (tooltip or popover)
  4. On mouseExited → hide it

  let area = NSTrackingArea(
      rect: button.bounds,
      options: [.mouseEnteredAndExited, .activeAlways, .inVisibleRect],
      owner: self
  )
  button.addTrackingArea(area)

  Notes:
  - .activeAlways works even when the app isn't frontmost (menu bar item)
  - .inVisibleRect auto-updates the rect if the button resizes
  - mouseEntered/mouseExited go to the owner (StatusBar inherits from NSObject)

--- Future-proofing ---

No concerns. The tracking area + event monitor pattern is standard AppKit.
If more modifier-based behaviors are added later (Cmd-hover, etc.), the
flag-checking can be refactored trivially at that point.
