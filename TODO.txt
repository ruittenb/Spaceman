OPTION + RIGHT CLICK: show spatial mini-map of spaces in a popup balloon
================================================================================

Status:   IDEA (2026-04-03)
Effort:   Medium — NSPopover + SwiftUI grid + slider, hover detection
Risk:     Low — additive, no impact on existing functionality

--- What it is ---

When the user hovers over the menu bar icon (optionally with Option held),
a popup balloon appears showing all spaces in a grid layout — a mini
Mission Control. The current space is highlighted. Clicking a cell switches
to that space.

The grid dimensions are controlled by a slider inside the popup itself.
The user sets one dimension (e.g. number of columns), and the other
follows from the total space count. For example, with 9 spaces and the
slider set to 3 columns, the grid is 3×3. With 8 spaces and 2 columns,
it's 2×4.

--- Why it's useful ---

The linear menu bar strip shows which space you're on, but not where it
sits relative to the others spatially. The grid gives that at a glance.

This makes "Current only" mode especially powerful for users with busy
menu bars: the menu bar icon is tiny (just one space indicator), but
hovering reveals the full spatial map. Minimal footprint, full awareness.

--- What the popup contains ---

  ┌───────────────────────────┐
  │  ┌─────┐ ┌─────┐ ┌─────┐  │
  │  │  1  │ │ [2] │ │  3  │  │
  │  └─────┘ └─────┘ └─────┘  │
  │  ┌─────┐ ┌─────┐ ┌─────┐  │
  │  │  4  │ │  5  │ │  6  │  │
  │  └─────┘ └─────┘ └─────┘  │
  │  ┌─────┐ ┌─────┐ ┌─────┐  │
  │  │  7  │ │  8  │ │  9  │  │
  │  └─────┘ └─────┘ └─────┘  │
  │                           │
  │  Columns: ──●────── 3     │
  └───────────────────────────┘

  - Each cell shows the space number, name, or emoji (matching the
    current display style setting)
  - Current space is highlighted (filled/colored)
  - Per-space colors are shown
  - Clicking a cell switches to that space
  - The slider at the bottom controls the number of columns (or rows)
  - Slider value is persisted in UserDefaults

--- Hover detection ---

Same mechanism as the original OPTION-HOVER idea:

  1. NSTrackingArea on statusBarItem.button for mouseEntered/mouseExited
  2. On mouseEntered: check NSEvent.modifierFlags.contains(.option)
     (or show unconditionally — TBD)
  3. If triggered → show NSPopover with the grid view
  4. On mouseExited → dismiss (with a small delay to allow entering the
     popover itself)

  let area = NSTrackingArea(
      rect: button.bounds,
      options: [.mouseEnteredAndExited, .activeAlways, .inVisibleRect],
      owner: self
  )
  button.addTrackingArea(area)

--- Grid rendering ---

SwiftUI LazyVGrid or manual VStack/HStack:

  let columns = Array(repeating: GridItem(.flexible()), count: numColumns)
  LazyVGrid(columns: columns, spacing: 4) {
      ForEach(spaces) { space in
          SpaceCell(space: space, isCurrent: space.isCurrentSpace)
              .onTapGesture { switchToSpace(space) }
      }
  }

--- Considerations ---

1. Trigger: Option-hover vs always-hover. Option-hover is safer (no
   accidental popups). Always-hover is more discoverable. Could be a
   preference toggle, or start with Option-hover.

2. Multi-display: Each display has its own spaces. The grid could show
   all displays (with separators) or just the current display's spaces.

3. Dismiss behavior: The popover should stay open while the cursor is
   inside it (not just while hovering the menu bar icon). Need to track
   mouseExited on both the button and the popover.

4. Slider persistence: Store the column count in UserDefaults via
   @AppStorage. Default could be auto-calculated: ceil(sqrt(spaceCount)).

5. Interaction with existing click behavior: Left-click still switches
   spaces via the menu bar icon directly. The popup is hover-only.
   Right-click still opens the settings menu. The popup doesn't replace
   either — it adds a third interaction mode.

OPTION-HOVER (original idea): show extra info on the menu bar icon
================================================================================

Status:   SUPERSEDED by HOVER GRID above (2026-04-03)
Effort:   Low — isolated change, entirely within StatusBar.swift
Risk:     None — additive, no architectural impact, easy to remove

Why superseded: The hover grid idea covers this use case and goes further.
The original idea of showing a tooltip or simple popover with space names
is a subset of the grid popup. Keeping this section for historical context.

--- What it does ---

When the user hovers over the menu bar icon while holding Option, show
additional detail about the spaces.

--- What to show (candidates) ---

  a) Space numbers and/or names — for users who use compact/number styles and
     occasionally want to see which space is which without opening the menu.
     Easiest.

  b) Space screenshots — tiny thumbnails of each space's content. Much cooler
     but much harder: needs CGWindowListCreateImage, Screen Recording
     permission, and image rendering. Different league of effort.

  c) Other ideas considered: focused app per space.

--- How to present it (candidates) ---

  a) Tooltip — simplest. Just set statusBarItem.button.toolTip to a string.
     Plain text only, no formatting control. Only works for (a) above.

  b) NSPopover — speech balloon like the help popovers in Preferences (which
     use SwiftUI .popover). The menu bar icon is AppKit, so this would use
     NSPopover with an NSHostingView to embed SwiftUI content. More effort
     but can show rich/formatted content, and the pattern already exists in
     the codebase (PreferencesView.swift:262, :421). Works for all options.

--- Hover detection (common to all approaches) ---

All in StatusBar.swift (~15-20 lines):

  1. NSTrackingArea on statusBarItem.button for mouseEntered/mouseExited
  2. On mouseEntered: check NSEvent.modifierFlags.contains(.option)
  3. If Option held → show the detail (tooltip or popover)
  4. On mouseExited → hide it

  let area = NSTrackingArea(
      rect: button.bounds,
      options: [.mouseEnteredAndExited, .activeAlways, .inVisibleRect],
      owner: self
  )
  button.addTrackingArea(area)

  Notes:
  - .activeAlways works even when the app isn't frontmost (menu bar item)
  - .inVisibleRect auto-updates the rect if the button resizes
  - mouseEntered/mouseExited go to the owner (StatusBar inherits from NSObject)

--- Future-proofing ---

No concerns. The tracking area + event monitor pattern is standard AppKit.
If more modifier-based behaviors are added later (Cmd-hover, etc.), the
flag-checking can be refactored trivially at that point.

RIGHT-CLICK DEAD ZONE: top 1-2px of menu bar icon
================================================================================

Status:   PARKED (2026-03-28)
Effort:   Low-Medium — NSEvent monitor + coordinate check
Risk:     Low — additive, but must avoid double-triggering with normal clicks

--- What it is ---

Commit c410b92 fixed left-clicks in the 1-2px gap above/below the menu bar
button by using NSEvent.mouseLocation instead of event.locationInWindow.
Right-clicks in that gap are NOT fixed: the button's sendAction(on:) never
fires because the click lands outside the NSStatusBarButton's hit area, so
handleClick() is never called.

--- Why it matters ---

Fitts's law: the topmost pixel of the screen is "infinitely tall" because the
cursor stops at the screen edge. Right-clicking the menu bar icon by slamming
the cursor to the top of the screen almost always lands in the 1-2px gap,
not on the button. This is the most natural way to right-click a menu bar
item, and it silently does nothing.

--- Approach ---

Add an NSEvent.addLocalMonitorForEvents(matching: .rightMouseDown) that:

  1. Checks if the click is within the status bar item's screen frame
     (statusBarItem.button.window.convertToScreen(button.frame))
  2. If so, shows the popup menu manually (same code as the existing
     right-click path in handleClick)
  3. Returns nil to consume the event (prevents double-triggering)

The monitor must coexist with the button's own sendAction — when the click
lands on the button proper, the button handles it and the monitor should
not interfere. A simple approach: always handle right-clicks via the monitor
and remove .rightMouseDown from sendAction(on:). This avoids any
double-trigger logic.

--- Why parked ---

The impact is real but narrow. Users who right-click precisely (cursor
already near the menu bar) are unaffected. Only the "slam to top edge"
gesture fails. Left-clicks (the primary interaction) already work in the
gap.

SLANTED ICON SHAPES: triangles and trapezoids
================================================================================

Status:   IDEA (2026-04-03)
Effort:   Medium — custom NSBezierPath rendering, hit testing changes
Risk:     Low — additive, new icon shape option alongside existing ones

--- What it is ---

A new decoration shape where icons have slanted/diagonal edges instead of
vertical ones. With short text (numbers), icons become triangles. With longer
text (names), they become trapezoids. The slant creates a distinctive,
dynamic look.

Numbers (triangles), without or with gaps:

      _______                   . _____ .    
     / \ 5 / \       or        / \\ 5 // \   
    / 4 \ / 6 \               / 4 \\ // 6 \  
    ‾‾‾‾‾‾‾‾‾‾‾               ‾‾‾‾‾ ' ‾‾‾‾‾  

Names (trapezoids) without or with gaps:

      ___________________                  ____ ________ _______   
     /    \ Five /       \      or        /    \\ Five //       \  
    / Four \    / Private \              / Four \\    // Private \ 
    ‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾              ‾‾‾‾‾‾‾‾ ‾‾‾‾ ‾‾‾‾‾‾‾‾‾‾‾ 

Icons can tessellate (share slanted edges) or have gaps between them, like
the existing rectangular icons do.

--- Rendering ---

NSBezierPath can draw arbitrary polygons, so the shape itself is
straightforward — just a trapezoid/triangle path instead of the current
NSBezierPath(roundedRect:). Would be a new case in IconStyle or IconShape.

For each icon, the path would be:
  - Top edge: shorter than bottom (or equal for rectangles)
  - Left edge: slanted inward from bottom-left to top-left
  - Right edge: slanted inward from bottom-right to top-right
  - The slant angle is constant; the top width depends on the text width

Text is centered within the shape. For trapezoids this is approximately
the center of the bounding rect; for narrow triangles the text sits in
the wider bottom portion.

--- Considerations ---

1. Hit testing: IconWidth currently uses rectangular left/right bounds.
   Slanted edges mean the actual clickable shape is a trapezoid. Two
   approaches:
     a) Keep rectangular hit zones (simpler, small visual mismatch at edges)
     b) Add diagonal hit testing (more accurate, more code)
   Option (a) is probably fine — the mismatch is 1-2px at the slant.

2. Interlocking vs gaps: The shapes can tessellate (each icon's right slant
   is the next icon's left slant) or have gaps. Gaps are simpler since icons
   remain independently rendered. Tessellation requires either rendering the
   whole strip as one composite path or handling overlap between adjacent
   icon images.

3. Two-row mode: Slanted shapes in two rows could alternate the slant
   direction per row (top row: /‾\, bottom row: \_/) for a zig-zag effect.
   Or both rows could slant the same way. Needs visual experimentation.
   Or this shape could be unavailable in two-row mode.

4. Active/inactive distinction: The existing fill/border distinction works
   naturally with trapezoid paths — filled trapezoids vs bordered trapezoids.

5. Fullscreen variant: Could use inverted slant (wider at top instead of
   bottom) as the contrasting shape, analogous to how rectangular↔pill works
   currently.

--- Implementation sketch ---

  - Add a new IconShape case (e.g. .slanted) or a separate slant parameter
  - In renderIcon(), when slanted: build an NSBezierPath with 4 points
    instead of using roundedRect
  - The slant inset = some fraction of the icon height (e.g. height * 0.3)
  - Top-left = (slantInset, height), top-right = (width - slantInset, height)
  - Bottom-left = (0, 0), bottom-right = (width, 0)
  - For bordered: stroke the path. For filled: fill the path
  - Text drawing unchanged (centered in bounding rect, close enough)

UNIFIED POPUP PANEL: replace NSMenu + Preferences window with a single panel
================================================================================

Status:   IDEA (2026-04-03)
Effort:   High — new UI architecture, replaces two existing systems
Risk:     Medium — touches all user-facing settings UI

--- KEY TRADE-OFF: speed vs. organization ---

The right-click menu exists because it's FAST. Changing icon size is two
actions: right-click → hover submenu → click. A tabbed panel would make
this three or four: right-click → click tab → open dropdown → select.
That's a regression for the most common quick-settings use case.

The Preferences window exists because it's ORGANIZED. Everything in one
place, clearly labeled, with room for complex controls.

Merging them optimizes for one goal at the expense of the other. If this
idea is pursued, mitigations for the speed regression:
  - Option-scroll for size changes remains the fastest path (no menu at all)
  - The panel could remember the last-used tab
  - The most-changed settings (size, spaces shown) could appear on the
    default Spaces tab alongside the space list, avoiding tab switches
  - Or: keep the right-click menu for quick settings AND the panel for
    deep settings. Accept the duplication as the cost of good UX in both
    contexts. The menu is fast because it's shallow; the panel is complete
    because it's deep. Different tools for different needs.

--- Motivation ---

Settings currently live in two places:
  1. The right-click NSMenu (Icon Size, Icon Text, Icon Shape, Row Layout,
     Spaces Shown — with checkmarks and submenus)
  2. The Preferences window (all settings, organized in four tabs)

This duplication means changes must be kept in sync. Adding a new setting
requires updating both the menu and the Preferences window. Some settings
are only in Preferences (font, icon widths, active/inactive style, colors,
backup/restore, shortcuts). The right-click menu is a subset.

--- The idea ---

Replace both with a single popup panel anchored to the menu bar icon:

  - Left-click on a space icon → switches spaces (unchanged)
  - Right-click on the menu bar icon → opens the popup panel

The panel is a borderless NSPanel (or NSPopover) containing a SwiftUI view
with tabs, similar to the current Preferences window but appearing as a
dropdown from the menu bar. Apps like Bartender, iStat Menus, and Dato
use this pattern.

--- Panel structure ---

  Tabs: [ Spaces | Appearance | General | Shortcuts ]

  Spaces tab:
    - List of spaces with names and checkmarks for the current space
    - Clicking a space switches to it (replaces the NSMenu space list)
    - Space name editing (currently in Preferences → Spaces)
    - Color assignment per space

  Appearance tab:
    - All appearance settings in one place (currently split between menu
      and Preferences → Appearance)
    - Icon size, widths, text, font, active/inactive style, rows, etc.

  General tab:
    - Launch at login, refresh, display ordering, backup/restore
    - Same as current Preferences → General

  Shortcuts tab:
    - Keyboard shortcut configuration
    - Same as current Preferences → Shortcuts

--- What goes away ---

  - The right-click NSMenu (replaced by the panel)
  - The separate Preferences window (merged into the panel)
  - The "Open Preferences" menu item (no longer needed)
  - The duplicate settings code in StatusBar.swift

--- What stays ---

  - Left-click space switching (unchanged)
  - Option-scroll to resize (unchanged)
  - AppleScript commands (unchanged)

--- Implementation approach ---

  1. Create a borderless NSPanel positioned below the status bar item
  2. Host the existing SwiftUI PreferencesView (or a refactored version)
     inside it via NSHostingView
  3. Add dismiss-on-click-outside behavior (NSEvent global monitor for
     mouseDown outside the panel)
  4. Add escape-to-dismiss
  5. Integrate the space list + switching into the Spaces tab
  6. Remove the NSMenu construction from StatusBar.swift
  7. Remove the separate PreferencesWindow

--- Considerations ---

  1. Panel width: The current Preferences window is ~500px wide. A menu
     bar popup that wide might feel oversized. May need a narrower layout
     or allow the panel to be wider than the status bar item.

  2. Space name editing: The Spaces tab currently has a scrollable list
     of text fields. This needs decent vertical space. A popup panel
     might need to be taller than a typical menu.

  3. Keyboard shortcut recording: The KeyboardShortcuts library shows a
     recording UI inline. This works in a window but might feel cramped
     in a popup. Needs testing.

  4. The "About" info (version, GitHub/Website links) needs a home —
     could go at the top of the panel or in a small footer.

  5. Quit button needs a home — bottom of the panel, always visible.

  6. Sparkle update UI currently uses the Preferences window as parent.
     Would need to adapt to the popup panel.

--- Why parked ---

This is a v2.0-level change. The current system works well. The
duplication is manageable. This idea is worth pursuing when there's
appetite for a bigger UI overhaul, not as a mid-release tweak.
